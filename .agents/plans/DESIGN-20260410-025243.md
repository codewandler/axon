# Design: Axon — Coding Agent Feature Set
**Date**: 2026-04-10  
**Author**: coder agent  
**Status**: Draft — awaiting approval  

---

## Context

This design addresses the gap between what Axon currently is (a structural graph of files, dirs,
git history, and Go symbols) and what a coding agent actually needs to reason about code with
confidence. It was informed by reading the full source tree before writing a single line.

### What already exists (do not re-implement)

| Feature | Status | Location |
|---|---|---|
| Go symbol indexing (struct, interface, func, method, field, const, var) | ✅ Done | `indexer/golang/indexer.go` |
| Reference nodes (`go:ref`) — usage tracking | ✅ Done | `indexer/golang/indexer.go:917` |
| Import reference tracking via `go:ref` (RefKindImport) | ✅ Done | `indexer/golang/indexer.go:1034` |
| Project detection indexer | ✅ Done | `indexer/project/indexer.go` |
| AQL query language (all 4 phases) | ✅ Done | `aql/`, `adapters/sqlite/aql.go` |
| Context gathering (`axon context`) | ✅ Done | `context/` |
| TUI (`axontui`) | ✅ Done | `cmd/axontui/` |

### What is missing (this design covers)

1. **Import Graph** — direct `package -[imports]-> package` edges
2. **Interface implementation edges** — `struct -[implements]-> interface`
3. **Test-to-source linkage** — test packages indexed and linked to source packages
4. **Watch / incremental mode** — `axon watch` command
5. **Semantic/vector search** — embedding storage and similarity queries
6. **Blast-radius CLI** — `axon impact <symbol>` command

## Feature 1 — Import Graph (Direct Package→Package Edges)

### Problem

Currently imports are tracked as `go:ref` nodes with `RefKindImport`. This means:
- Each import site creates a `go:ref` node
- To find "what does package A import?" you must query: 
  `SELECT * FROM nodes ... go:ref ... RefKindImport` — expensive and indirect
- There are no direct `go:package -[imports]-> go:package` edges

An agent answering "what would break if I change package X?" needs to walk the reverse import
graph: "who imports X?" One edge traversal should be enough.

### Design

**During `indexPackage()`** (in `indexer/golang/indexer.go`), after creating the `go:package`
node, iterate `pkg.Imports` (already available from `go/packages` with `NeedImports`) and emit
one `imports` edge per imported package that is within the same module.

```
go:package (importer) -[imports]-> go:package (imported)
```

The `imports` edge type already exists in `types/edges.go` (`EdgeImports = "imports"`).

**Scope**: Only emit edges for intra-module imports (same module path prefix). Cross-module
packages are external and won't have node representations.

**URI construction**: The imported package's URI follows the existing pattern:
`{moduleURI}/pkg/{importPath}` — same as in `indexPackage()`.

**New `PackageData` fields**:
```go
ImportPaths []string `json:"import_paths,omitempty"` // direct import paths
```

**AQL query this enables**:
```aql
-- What does the aql package import?
SELECT imported FROM (pkg:go:package)-[:imports]->(imported:go:package)
WHERE pkg.data.import_path = 'github.com/codewandler/axon/aql'

-- Who imports adapters/sqlite?
SELECT importer FROM (importer:go:package)-[:imports]->(pkg:go:package)
WHERE pkg.data.import_path LIKE '%adapters/sqlite'
```

## Feature 2 — Interface Implementation Edges

### Problem

The graph knows that `Storage` is an `go:interface` and `sqlite.Storage` is a `go:struct`.
It does NOT know that `sqlite.Storage` implements `graph.Storage`. This is the single most
useful structural relationship in a Go codebase.

An agent answering "show me all implementations of Storage" currently has no path.

### Design

**New edge type**: `EdgeImplements = "implements"` in `types/edges.go`.

**Where to emit**: In a new `indexImplementations()` function called from `indexPackage()`,
after all types in the package have been indexed. Use `pkg.TypesInfo` and `pkg.Types.Scope()`
to iterate over all named types in the package, then use `go/types.Implements()` to check each
struct against each interface in the module.

**Algorithm**:
1. Collect all interfaces in the module scope (from already-indexed nodes, resolved via URI)
2. For each struct in the current package, call `types.Implements(structType, iface)` or
   `types.Implements(types.NewPointer(structType), iface)`
3. If true, emit: `struct_node -[implements]-> interface_node`

**Scope**: Only check interfaces within the same module (same moduleURI prefix) to avoid
expensive cross-module type resolution.

**New edge registration** in `types/golang.go` `RegisterGoTypes()`:
```go
r.RegisterEdgeType(graph.EdgeSpec{
    Type:        EdgeImplements,
    Description: "Struct implements interface",
    FromTypes:   []string{TypeGoStruct},
    ToTypes:     []string{TypeGoInterface},
})
```

**AQL query this enables**:
```aql
-- Who implements graph.Storage?
SELECT impl FROM (impl:go:struct)-[:implements]->(iface:go:interface)
WHERE iface.name = 'Storage'

-- What interfaces does sqlite.Storage implement?
SELECT iface FROM (s:go:struct)-[:implements]->(iface:go:interface)
WHERE s.name = 'Storage' AND s.data.import_path LIKE '%sqlite%'
```

## Feature 3 — Test-to-Source Linkage

### Problem

`packages.Load(cfg, "./...")` currently has `Tests: false`. Test files are invisible to the
Go indexer. An agent cannot answer "is there a test for this function?" or "what does this
test cover?".

### Design

**New `go:package` field**: `IsTest bool` already exists in `PackageData` but is never set.

**Change `cfg.Tests` to `true`** in `indexPackage()`. Go's `go/packages` loader returns test
packages as `_test` suffixed variants (e.g., `aql_test`).

**New edge type**: `EdgeTests = "tests"` (test_package → source_package).

Emit: `test_package -[tests]-> source_package` when a `_test` package has the same import
path base as an existing non-test package (standard Go convention).

**Test file linkage**: For each `go:ref` node at a position inside a `_test.go` file, we
already have the position. No additional nodes needed — the test package node is sufficient.

**`PackageData` update**:
```go
IsTest      bool     `json:"is_test,omitempty"`
TestFor     string   `json:"test_for,omitempty"` // import path of package being tested
```

**AQL query this enables**:
```aql
-- Is there a test for package aql?
SELECT test FROM (test:go:package)-[:tests]->(pkg:go:package)
WHERE pkg.data.import_path = 'github.com/codewandler/axon/aql'

-- All untested packages
SELECT pkg FROM nodes pkg WHERE pkg.type = 'go:package'
  AND NOT EXISTS (SELECT 1 FROM (t:go:package)-[:tests]->(pkg))
  AND NOT pkg.data.is_test
```

## Feature 4 — Watch / Incremental Mode

### Problem

`axon init` re-indexes everything from scratch. After editing a file, the graph is stale.
A coding agent making changes needs the graph to reflect reality in near-real-time.

### Design

**New command**: `axon watch [path]`

**Strategy**: Use `fsnotify` (already likely a transitive dep, or add it) to watch the
indexed directory. On file-change events, re-index only the affected subtree.

**Implementation approach**:
- Add `func (a *Axon) Watch(ctx context.Context, path string, opts WatchOptions) error`
  to `axon.go`
- Internally: start `fsnotify.Watcher`, debounce events (100ms window), then call
  `IndexWithOptions()` with the changed file's parent directory as the root
- The existing generation-based GC already handles stale node removal

**`WatchOptions`**:
```go
type WatchOptions struct {
    IndexOptions
    Debounce    time.Duration // default: 100ms
    OnReindex   func(path string, result *IndexResult) // optional callback
}
```

**CLI**:
```
axon watch           # Watch current directory
axon watch ./src     # Watch specific subtree
axon watch --quiet   # No per-change output
```

**Out of scope for v1**: Watching individual files (only directory-level re-indexing).
Full file-level incremental indexing requires a more complex "dirty node" tracking system,
deferred to a future design.

## Feature 5 — Semantic / Vector Search

### Problem

AQL matches names, types, labels — structural queries. A coding agent needs to ask:
- "Find code that handles token budget overflow"
- "What does error recovery look like in this codebase?"

These are semantic queries. No amount of `LIKE` or `GLOB` can answer them.

### Design

This is the most significant new feature. It has two sub-parts: **embedding storage** and
**similarity search**.

#### 5a — Embedding Storage

**New column on the `nodes` table**: `embedding BLOB` — SQLite can store float32 arrays as
binary blobs. The `graph.Storage` interface gains:

```go
PutEmbedding(ctx context.Context, nodeID string, embedding []float32) error
GetEmbedding(ctx context.Context, nodeID string) ([]float32, error)
FindSimilar(ctx context.Context, query []float32, limit int, filter *NodeFilter) ([]*NodeWithScore, error)
```

**New `NodeWithScore` type**:
```go
type NodeWithScore struct {
    *Node
    Score float32
}
```

**Cosine similarity** is implemented in Go (not in SQLite). `FindSimilar` loads all embeddings
into memory, computes cosine similarity, returns top-K. This is viable for codebases up to
~100k nodes. HNSW or SQLite-VSS is a future optimization.

#### 5b — Embedding Indexer

**New indexer**: `indexer/embeddings/indexer.go`

- Implements `PostIndexer` interface (runs after all other indexers flush)
- Iterates over all `go:func`, `go:struct`, `go:interface`, `md:section`, `fs:file` nodes
- For each node, builds a text representation: `name + doc + signature + labels`
- Calls an embedding provider (configurable: OpenAI `text-embedding-3-small`, local Ollama
  `nomic-embed-text`, or a stub for testing)
- Stores embedding via `PutEmbedding()`

**Provider interface**:
```go
type EmbeddingProvider interface {
    Embed(ctx context.Context, text string) ([]float32, error)
    Dimensions() int
}
```

Built-in providers:
- `NullProvider` — returns zero vector (no-op, for testing)
- `OllamaProvider` — calls `http://localhost:11434/api/embeddings`
- `OpenAIProvider` — calls OpenAI API

**Configuration** via `axon.Config`:
```go
EmbeddingProvider embeddings.Provider // nil = disabled
```

#### 5c — CLI command: `axon search`

Already exists (`cmd/axon/search.go`). Extend to support semantic mode when embeddings are
available:

```
axon search "handles token budget overflow"    # keyword search (current)
axon search --semantic "token budget overflow" # vector similarity search (new)
axon search --semantic --type go:func "..."    # scoped semantic search
```

## Feature 6 — Blast Radius CLI (`axon impact`)

### Problem

After implementing Features 1–3, all the graph data exists. But a coding agent still needs
to *ask the question* efficiently. A dedicated command makes the most common question trivial.

### Design

**New command**: `axon impact <symbol>`

Given a symbol name (or partial URI), walk the reverse dependency graph:
1. Find the node matching the symbol
2. Find all `go:ref` nodes that reference it (reverse `references` edges)
3. For each ref, find its containing package
4. Find all packages that import those packages (reverse `imports` edges)
5. Report a ranked list: "if you change X, these files/packages are affected"

```
$ axon impact Storage

Impact: graph.Storage (go:interface)

Direct references (17):
  adapters/sqlite — Storage.PutNode, Storage.GetNode, ... (12 refs)
  cmd/axon       — Storage.Query (2 refs)
  context        — Storage.FindNodes (3 refs)

Packages importing affected packages (3):
  axon           — imports adapters/sqlite
  cmd/axon       — imports adapters/sqlite
  context        — imports graph
```

**Implementation**: Pure AQL + post-processing. No new storage features needed once
Features 1–2 are done.

## Acceptance Criteria

| Feature | Done when... |
|---|---|
| Import Graph | `axon query "(p:go:package)-[:imports]->(q:go:package) SELECT p, q"` returns edges |
| Implements edges | `axon query "(s:go:struct)-[:implements]->(i:go:interface) SELECT s, i"` returns results |
| Test linkage | `axon query "(t:go:package)-[:tests]->(p:go:package) SELECT t, p"` returns results |
| Watch mode | `axon watch` keeps graph updated as files change; `axon find` reflects changes |
| Semantic search | `axon search --semantic "token budget"` returns ranked results |
| Blast radius | `axon impact Storage` prints a structured impact report |

---

## Out of Scope (this design)

- Other language symbol indexers (TypeScript, Python, Rust) — Go is the priority
- HNSW or SQLite-VSS for scalable vector search — cosine scan is sufficient for now
- File-level incremental indexing in watch mode — directory re-index is good enough for v1
- Cross-module import graph — requires fetching external module sources
- Generic type parameter tracking — deferred; current type alias skipping is acceptable

---

## Risks & Dependencies

| Risk | Mitigation |
|---|---|
| `types.Implements()` is slow on large codebases | Cache interface types; only check intra-module structs |
| Embedding API cost/latency | `NullProvider` default; lazy embedding (only on `--semantic` flag) |
| `fsnotify` dependency | Already likely transitive; add directly to `go.mod` if not |
| `Tests: true` inflates package count | Filter `_test` packages from stats/tree output unless requested |
| SQLite BLOB column migration | Add migration path in `adapters/sqlite/sqlite.go` schema init |
